# LAUNCH_BLOCKERS — 상업 출시 차단 위험

> PRD §22 step 7 산출물. 본 문서는 WorshipScore AI의 **상업 출시(commercial launch)를 막는 차단 위험(launch blockers)** 을 우선순위로 정리한다.
> 작성 기준일: 2026-06-16 · 현재 진행 단계: Milestone 1 완료(결정적 핵심 체인 실증).
> 각 항목은 위험(risk)·영향(impact)·현상태(status)·해소 조건(exit criteria)·담당 Milestone으로 구성하며, PRD §16.2 출시 게이트와 연결한다.
> 사실에 근거하지 않은 수치는 'TBD(추후 측정)'로 표기한다.

---

## 0. 요약 — 출시 게이트와 차단 위험 매핑

PRD §16.2가 정의하는 **출시 게이트(launch gate)** 는 다음과 같다. 본 Milestone 1 단계에서는 게이트의 어느 항목도 정식으로 충족되었다고 선언할 수 없다(핵심 체인의 결정적 동작만 실증됨).

| # | PRD §16.2 출시 게이트 기준 | 현 단계 충족 여부 | 직접 연관 Blocker |
|---|---|---|---|
| G1 | 치명적 음악 오류(critical musical error) 0건 | 미충족 — OMR 미검증, 측정 불가 | LB-01 |
| G2 | 처리 완료율(completion rate) ≥ 90% | 미충족 — 업로드/job 파이프라인 미구현 | LB-06 |
| G3 | 검수(review) 5분 이내 | 미충족 — 검수 UI 미구현 | LB-12 |
| G4 | PPTX 열기 실패율 < 0.1% | 미충족 — 실 PowerPoint 라이브 미검증 | LB-03 |
| G5 | 요소 잘림(clipping) 0건 | 부분 충족 — Milestone 1에서 contain-fit으로 잘림 0 실증, 일반화는 TBD | LB-03 |
| G6 | 조직 간 노출(cross-org exposure) 0건 | 미충족 — 멀티테넌트 격리 미구현 | LB-04 |
| G7 | 수정 유실(edit loss) 0건 | 미충족 — ScoreIR 영속화/immutable revision 미구현 | LB-11 |

> 게이트는 모두 **AND 조건**으로 간주한다. 단일 Blocker라도 미해소 시 상업 출시 불가.

## 2. 차단 위험 상세

### LB-01 — OMR 정확도 미검증
- **위험**: 악보 이미지/PDF에서 ScoreIR을 생성하는 OMR(Optical Music Recognition) 경로의 정확도가 한 번도 측정되지 않았다. 핵심 제품 가치(악보형 PPT 자동 생성)의 입력 품질이 검증되지 않은 상태다.
- **영향**: 치명적 음악 오류(잘못된 음높이·리듬·조성)가 그대로 슬라이드에 출력될 수 있어 PRD §16.2 G1(치명적 음악 오류 0)을 직접 위반한다. 예배 현장에서의 오류는 신뢰도에 직접 타격.
- **현상태**: `packages/core` `src/types/providers.ts`에 `OMRProvider` 인터페이스만 정의됨. 실제 엔진 연동·골든셋·정확도/비용/시간 측정은 미착수. Milestone 1 범위에서 명시적으로 제외됨.
- **해소 조건**:
  - 골든셋(golden set) 구축 및 후보 엔진 spike 완료.
  - 치명적 음악 오류율 측정값 확보(목표 0건, 현재 측정값 TBD).
  - 정확도·비용·처리시간 벤치마크 문서화(TBD).
- **담당 Milestone**: M3(OMR 기술검증).

### LB-02 — Audiveris AGPL 법무 미승인
- **위험**: 유력 OMR 후보인 Audiveris는 **AGPL-3.0** 라이선스다. SaaS 형태 배포 시 AGPL 네트워크 조항(소스 공개 의무)이 적용될 수 있다. "별도 worker 분리로 AGPL 의무가 자동 해소된다"는 가정은 채택하지 않는다.
- **영향**: 미승인 상태에서 프로덕션 경로에 편입되면 상업 라이선스 위반/소스 공개 강제 등 법적 리스크가 발생한다. 출시 전 반드시 법무 게이트 통과 필요.
- **현상태**: Audiveris는 **현재 의존성에 추가되지 않음**. 법무 승인 전 프로덕션 경로 진입 금지 게이트가 적용 중. (실행 환경에 Java 없음 → 현 단계 Audiveris/MuseScore CLR 사용 불가, 단 이는 무관한 부수 사실.)
- **해소 조건**:
  - 법무 검토 및 승인 문서 확보(또는 비-AGPL 대체 엔진 채택).
  - 채택 시 worker 격리 아키텍처의 AGPL 의무 충족 여부 명문 검토(자동 해소 가정 금지).
  - SBOM 및 라이선스 승인 체계(LB-14)와 연동.
- **담당 Milestone**: M3(AGPL 게이트) / M9(상업화 라이선스 승인).

### LB-03 — PPTX 실 PowerPoint 호환성 라이브 미검증
- **위험**: 생성된 PPTX가 실제 Microsoft PowerPoint에서 "수리 경고(repair prompt) 없이" 열리는지 라이브로 확인되지 않았다.
- **영향**: 열기 실패 또는 수리 경고는 PRD §16.2 G4(열기 실패율 < 0.1%)를 위반하고 사용자 신뢰를 직접 훼손한다. 요소 잘림(G5)도 일반 입력에 대해서는 미보장.
- **현상태**:
  - Milestone 1에서 `out/worship-score-sample.pptx`(213,481 bytes, 4슬라이드) 생성 및 `validatePptxOoxml`의 13개 OOXML 구조 항목 전부 통과(zip_loadable … each_slide_has_image).
  - PNG는 안전여백 박스 contain-fit으로 배치되어 해당 샘플에서 **잘림 0** 실증.
  - 그러나 **실 PowerPoint 라이브 테스트는 미실행**(개발 머신에 Office 없음). 일반 입력에 대한 잘림 0 일반화는 미검증.
- **해소 조건**:
  - 실 PowerPoint(데스크톱/웹/모바일) 열기 실패율 < 0.1% 달성(측정값 TBD).
  - 대표 입력 코퍼스에서 잘림 0 유지 검증(측정값 TBD).
  - 수동/CI 절차로 라이브 호환성 추적 체계 수립.
- **담당 Milestone**: M7(발표구성/PPT) / M9(CI 라이브 검증 절차).

### LB-04 — 멀티테넌트 격리 미구현
- **위험**: 조직(tenant) 간 데이터 격리가 구현되지 않았다. 현재 단계에는 DB·인증·조직 모델 자체가 없다(Milestone 1은 결정적 체인만 구현, 웹/DB/UI 미도입).
- **영향**: 조직 간 노출(cross-org exposure)은 PRD §16.2 G6(조직 노출 0)을 직접 위반하며 상업 SaaS의 가장 치명적인 신뢰 사고에 해당한다.
- **현상태**: 미구현. DB·테넌시 경계·접근 제어 모두 미도입.
- **해소 조건**:
  - 테넌트 경계가 강제되는 데이터 모델·접근 제어 구현.
  - 조직 간 노출 0건을 보장하는 격리 테스트(부정 테스트 포함) 통과(결과 TBD).
- **담당 Milestone**: M2(업로드/객체저장 시 테넌시 도입) / M8(조직 라이브러리/통합 PPT).

### LB-05 — 저작권 신고/삭제(notice & takedown) 절차 미구현
- **위험**: 찬양 악보는 저작권 보호 대상이 많다. 권리자 신고 접수·콘텐츠 삭제(takedown)·재발 방지 절차가 없다.
- **영향**: 저작권 침해 신고에 대응 불가 시 법적 책임 및 서비스 중단 리스크. 상업 출시의 법적 전제 조건.
- **현상태**: 미구현. M2에 "권리확인(rights check)"이 범위로 잡혀 있으나 신고/삭제 운영 절차는 별도.
- **해소 조건**:
  - 업로드 시 권리확인 게이트 동작(M2).
  - 신고 접수 채널·삭제 처리 절차·기록 보존 문서화 및 구현.
- **담당 Milestone**: M2(권리확인) / M9(상업화 운영 절차).

### LB-06 — 결제/사용량 원장(billing & metering ledger) 미구현
- **위험**: 과금·사용량 측정 원장이 없다. 상업 SaaS의 수익화 기반이 부재.
- **영향**: 유료 출시 불가. 또한 사용량/완료율 측정 부재는 G2(완료율 ≥ 90%) 모니터링을 간접적으로 어렵게 한다.
- **현상태**: 미구현.
- **해소 조건**:
  - 과금 모델·사용량 원장·정합성(idempotent metering) 구현 및 검증(결과 TBD).
- **담당 Milestone**: M9(상업화 — 과금).

### LB-07 — 백업/복구(backup & restore) 미문서화
- **위험**: 데이터 백업·복구 절차가 문서화·구현되지 않았다(현 단계에는 영속 데이터 자체가 없음).
- **영향**: 데이터 유실 시 복구 불가. 수정 유실(G7)·운영 신뢰성과 직결.
- **현상태**: 미문서화·미구현. 영속화(LB-11)가 선행되어야 의미를 가짐.
- **해소 조건**:
  - 백업/복구 절차 문서화 및 복구 리허설(restore drill) 성공 기록(결과 TBD).
- **담당 Milestone**: M9(상업화 — 백업/복구).

### LB-08 — MusicXML XSD 엄격검증 미구현
- **위험**: 생성된 MusicXML 4.0에 대한 W3C/표준 XSD 스키마 엄격검증이 없다.
- **영향**: 비표준 MusicXML이 하류 도구(타 악보 소프트웨어·OMR 왕복)에서 거부될 수 있고, 잠재적 음악 데이터 오류(G1 간접)를 조기에 잡지 못한다.
- **현상태**: 현재는 **구조 검증 + Verovio 6.2.0 라운드트립(로드 성공)** 으로 대체. `packages/core` `src/musicxml/`의 직렬화는 결정적(동일 머신에서 11806 bytes 재현 확인). XSD 번들 도입은 추후.
- **해소 조건**:
  - MusicXML 4.0 XSD 번들 도입 및 직렬화 출력의 스키마 적합성 검증 통과(결과 TBD).
- **담당 Milestone**: M4(ScoreIR 영속화/검증 강화).

### LB-09 — PPTX 바이트 단위 결정성 미보장
- **위험**: PPTX 출력이 바이트 단위로 결정적이지 않다(pptxgenjs 4.0.1이 생성 타임스탬프를 삽입).
- **영향**: 동일 입력에 대한 재현성·감사(audit)·캐시 무효화 정확도에 제약. 출시 게이트 직접 위반은 아니나 감사·재현성 요건에 영향.
- **현상태**: 미보장. (대조적으로 ScoreIR→MusicXML, ScoreIR→SVG/PNG는 동일 머신에서 결정적임이 검증됨 — PNG 4장 2400×314px 결정적 확인.)
- **해소 조건**:
  - PPTX 정규화(타임스탬프 고정/제거 등) 후 바이트 단위 재현성 확보(결과 TBD).
- **담당 Milestone**: M7(템플릿/PPT) / M9(감사 요건).

### LB-10 — 가사 폰트 크로스-머신 결정성 미보장
- **위험**: 가사 텍스트 렌더링이 머신 간 결정적이지 않다(@resvg/resvg-js 2.6.2가 시스템 폰트를 사용).
- **영향**: 머신·환경에 따라 한글 가사의 글꼴/메트릭이 달라져 레이아웃 일관성과 잘림 위험(G5 간접)에 영향.
- **현상태**: 미보장. Milestone 1 샘플에서는 한글 가사가 정확히 렌더됨을 시각 확인했으나, 폰트 임베드/고정은 미적용.
- **해소 조건**:
  - 폰트 임베드 또는 고정 폰트셋 도입으로 크로스-머신 렌더 일관성 확보(결과 TBD).
- **담당 Milestone**: M7(발표구성/PPT 렌더링).

### LB-11 — ScoreIR 영속화/immutable revision 미구현
- **위험**: ScoreIR의 영속화·불변 리비전(immutable revision)·property test 강화가 미구현(현 단계는 인메모리 fixture 중심).
- **영향**: 사용자 수정 결과가 안전하게 보존·버전 관리되지 않으면 수정 유실이 발생해 PRD §16.2 G7(수정 유실 0)을 위반한다.
- **현상태**: 미구현. `SCOREIR_SCHEMA_VERSION='scoreir-0.1.0'` 타입·zod 스키마·검증 규칙(12개)은 존재하나 영속 계층은 없음.
- **해소 조건**:
  - 불변 리비전 저장 모델 구현 및 수정 유실 0 보장(부정 테스트 포함, 결과 TBD).
  - property-based test 강화.
- **담당 Milestone**: M4(영속화/immutable revision).

### LB-12 — 검수 UI(3열) 미구현 → 검수 5분 게이트 측정 불가
- **위험**: 사람 검수를 위한 3열 검수 UI가 없다(웹/UI 미도입).
- **영향**: PRD §16.2 G3(검수 5분 이내)를 측정·달성할 수단이 없다. 검수 없이는 G1(음악 오류 0) 보증도 사실상 불가.
- **현상태**: 미구현.
- **해소 조건**:
  - 3열 검수 UI 구현 및 대표 시나리오 검수 소요 ≤ 5분 측정(측정값 TBD).
- **담당 Milestone**: M6(검수 UI).

### LB-13 — 런타임 AI 검증·프롬프트 인젝션 방어 미구현
- **위험**: 런타임 AI 검증(model provider·tool contracts·structured output)과 프롬프트 인젝션(prompt injection) 방어가 미구현.
- **영향**: AI 경로가 신뢰 가능한 구조화 출력을 보증하지 못하면 음악 오류(G1 간접)·보안 사고로 이어질 수 있다.
- **현상태**: 미구현. (현 Milestone 1 체인은 결정적 로직 중심으로 AI 런타임 미포함.)
- **해소 조건**:
  - structured output 계약·도구 호출 검증·인젝션 방어 구현 및 보안 검토 통과(결과 TBD).
- **담당 Milestone**: M5(런타임 AI 검증).

### LB-14 — 감사 로그/SBOM/라이선스 승인 체계 미구현
- **위험**: 감사 로그(audit log)·SBOM·제3자 라이선스 승인 체계가 없다.
- **영향**: 상업 운영·컴플라이언스·LB-02(Audiveris AGPL) 게이트 운용의 기반이 부재. 사고 추적·법적 대응 불가.
- **현상태**: 미구현. 제3자 라이선스 인벤토리는 사실로 확보됨(아래 부록 참조).
- **해소 조건**:
  - 감사 로그 적재, SBOM 자동 생성, 라이선스 승인 워크플로 구축(결과 TBD).
- **담당 Milestone**: M9(상업화 — 감사/SBOM/라이선스 승인).

## 3. 현재 충족된 사항(출시 게이트 진전으로서)

차단 위험과 별개로, Milestone 1에서 **결정적 핵심 체인이 실증**되어 일부 게이트의 부분 근거를 확보했다. 이는 게이트 통과 선언이 아니라 진전 기록이다.

- 체인 전 구간 동작: ScoreIR → zod schema → `validateScore`(blocking 0/warning 0) → MusicXML 4.0(11806 bytes, 결정적) → Verovio 6.2.0(SVG+PNG) → slide plan(4슬라이드) → PptxGenJS PPTX(213,481 bytes) → OOXML 13항목 전부 통과 → preview PNG.
- 잘림 0 실증(샘플): PNG contain-fit 배치로 요소 잘림 0(샘플 한정, G5 부분 근거).
- 시각 확인: 슬라이드 PNG에 높은음자리표·4/4 박자·멜로디 음표·마디선·한글 가사 정확 렌더(악보형 PPT 실증).
- 테스트: vitest 전체 통과(`pnpm test`)(core 21 + pipeline fixture 5 + e2e 1), `tsc -p tsconfig.json` 타입체크 clean.

> 한계 명시: 위 결정성은 **동일 머신** 기준이며, PPTX 바이트 결정성(LB-09)·폰트 크로스-머신 결정성(LB-10)·실 PowerPoint 호환성(LB-03)은 별도로 미보장이다.
